What Goes Wrong in Large C++ Codebases?

❌  Changed one header file
    → entire codebase rebuilds (10+ min)

❌  Want to write a unit test
    → must link 30 unrelated libraries

❌  Cyclic dependency: A needs B, B needs A
    → neither can be tested alone

❌  "Reuse" a utility
    → it drags half the application along

Root Cause

These are not compiler problems.
They are physical design problems.

Physical Design = What Most Books Skip

Most developers learn logical design — classes, patterns, algorithms. Almost nobody teaches physical design — files, headers, dependencies.

Refactoring Feels Like Surgery

Touching anything breaks everything — that's not a skill problem. It's a structural problem caused by unchecked physical dependencies.

The Hidden Cost

No discipline around physical dependencies leads to pain that grows slowly — then all at once.

Physical Design Determines:

  • How long it takes to compile
  • Whether components can be tested independently
  • How much code you must link just to use one function
  • Whether you can reuse a library without dragging everything else

Lakos's Central Insight

"Physical design is not just about files and directories. It is about controlling the envelope of dependencies — both compile-time and link-time."

Both Are Required

Most software engineering books focus only on logical design and ignore physical design. This is a critical omission for large-scale systems.

One Without the Other Fails

Perfect logical design with poor physical design → unmaintainable codebase. The two must be designed together.

Even "application" code contains layers that ARE library software. Your date utilities, event system, string helpers — design them like libraries or pay the debt later.

The Trap

Software that starts as "just an application" often gets used as a library. Retrofitting good physical design is very expensive.

Rule of Thumb

Any code meant to outlive its first consumer should follow library software discipline.

The entire Lakos methodology is about accumulating software capital, not software debt. Every decision — file names, include discipline, dependency rules — serves this goal.

The return on software capital investment grows super-linearly. Each new component can leverage all prior components. Invest in physical design discipline early — it pays the largest dividends.

Vigilance Required

Physical design quality degrades without active vigilance. Cyclic dependencies creep in gradually. Regular architectural audits and enforced design rules are necessary to maintain quality.

Ch 0 — Software Capital

Collaborative → Reusable → Hierarchically Reusable

Collaborative Software

Pieces that only work together — like a cracked plate. Fits perfectly, but you cannot take one piece out of the set. No independent reuse possible.

Reusable Software

A single component with a clean interface, usable in isolation. Independent deployment is possible.

Hierarchically Reusable ← The Goal

Lower-level components reused to build higher-level ones, which are reused to build even higher-level ones. Each level independently testable and releasable.

Application Layer (uses all below) ↑ Calendar Subsystem (uses Date + String) ↑ Date Library (uses Integer Math) ↑ Integer Math (uses nothing internal)

Only Possible With Acyclic Deps

Hierarchical reuse requires acyclic dependencies. You can never have "wall A depends on wall B depends on wall A" — the structure collapses.

The Building Analogy

Bricks = Components. Walls = Packages. Building = Package Group. A brick can be used anywhere. You cannot reuse half a wall.

Ch 0 — Hierarchical Reuse

Malleable vs. Stable Software

Malleable Software (Applications)

  • Expected to change frequently with business needs
  • Can afford more coupling internally
  • Users are end users, not developers
  • Feature velocity is the primary goal

Stable Software (Libraries)

  • Must evolve without breaking existing clients
  • Requires strict physical design discipline from day one
  • Users are other developers who build on top of it
  • Stability and reusability are the primary goals

The Classic Trap

Software designed as "malleable" often gets used as a library. If designed carelessly, retrofitting good physical design is expensive — and frequently not done, leading to permanent technical debt.

The Difference Is Discipline

The same techniques apply to both. The difference is the degree of rigor and the extent to which design rules are enforced and maintained over time.

Design for Stability Early

Design every reusable component as if it will be a library. The cost of good design upfront is small. The cost of bad design at scale is enormous.

Chapter 1: The Fundamental Unit — The Component

// A component = one .h + one .cpp pair (same base name)
my_stack.h  +  my_stack.cpp  =  one component

// .h  = the CONTRACT — what clients see and depend on
// .cpp = the IMPLEMENTATION — clients never include this

Why the Component?

  • Automated dependency extraction works
  • Fine-grained unit testing becomes possible
  • Independent deployment is achievable
  • Physical uniformity enables tooling

The Brick Analogy

The component is to C++ code what a brick is to a building. You can reuse a brick anywhere. You cannot reuse half a wall.

CP2 — .cpp starts by including its own .h (enforces self-sufficiency)
// my_stack.cpp
#include <my_stack.h>   // ← ALWAYS FIRST — tests CP1
#include <stdexcept>    // other deps come after

void MyStack::push(int val) {
    d_data.push_back(val);
}

Free Enforcement of Self-Sufficiency

If the .h is missing a dependency, the .cpp fails to compile — because it includes the header first. No extra tooling needed.

Ch 1 — The Component

How C++ Programs Are Built

my_stack.cpp
    │
    ↓ [preprocessor: expand #includes and macros]
    │
translation unit (.i)   ← one giant merged file
    │
    ↓ [compiler: parse, optimize, machine code]
    │
object file (.o)   ← .text | .data | .bss | symbols
    │
    ↓ [linker: resolve symbols from .o + .a archives]
    │
executable / shared library

What #include Really Does

The preprocessor literally pastes the entire contents of a header into your translation unit — before the compiler sees a single line of your code.

The Compilation Tax

Every #include you add to a header is a compilation tax paid by every file that includes that header — and every file that includes those files.

The Linker's Perspective

Object files contain machine code plus a symbol table of what they provide and what they need. The linker resolves all unresolved symbols across the set of object files it's given.

Why Build Times Matter

A 10-minute rebuild for a one-line change is not a hardware problem. It's a physical design problem — too many things depending on too many headers.

Ch 1 — Build Process

Object Files and Library Archives

SectionContents
.textMachine code (function bodies)
.dataInitialized global / static data
.bssUninitialized global / static data
Symbol tableNames defined here + names needed from elsewhere

Static Library = Archive of .o Files

A .a file is just a bundle of .o files. The linker searches it for symbols needed by already-processed objects.

// library.a contains: math.o  string.o  graphics.o  network.o
// You use ONE function from math.o
→ Linker pulls in ALL of math.o
→ But NOT string.o, graphics.o, or network.o

The Critical Implication

The linker pulls in entire .o files, not individual functions. 50 unrelated functions in one .cpp means using any one forces all 50 on every user.

Component Granularity = Link-Time Footprint

Well-designed components contain tightly related functionality. When a client links your component, they get exactly what they need — no more, no less.

Ch 1 — Linker Mechanics

Declarations vs. Definitions & The ODR

// Declaration — a PROMISE: "this exists somewhere"
extern int g_counter;        // declared, not defined
class Date;                   // forward declaration
void processEvent(int type);  // declared

// Definition — "HERE it actually is"
int g_counter = 0;           // defined (storage allocated)
class Date { int d_year, d_month, d_day; };  // defined
void processEvent(int type) { /* body */ }   // defined

The One Definition Rule (ODR)

Declare as many times as needed. Define exactly once across the whole program. Violating this causes linker errors or silent undefined behavior.

Three Linkage Types

LinkageVisible InUse For
ExternalAll translation unitsPublic API (functions, classes)
InternalOne .cpp onlyImplementation helpers
NoneLocal scopeLocal variables
 
namespace {            // anonymous namespace = internal linkage
    void helperOnlyThisFile() { /* invisible outside */ }
}

Forward Declarations Are Powerful

Tells the compiler "this type exists" without requiring the full definition. Enables pointer/reference members without #include.

Ch 1 — Declarations & Linkage

Internal vs. External Linkage in Practice

❌ Never define non-inline functions in headers
// my_widget.h
void helper() { /* ... */ }  // ODR violation!
// Every .cpp that includes this gets its own copy
// → multiple definitions → linker error or silent UB
✅ Internal linkage for .cpp-only helpers
// my_widget.cpp
namespace {
    void helperOnlyThisFile() { /* completely hidden */ }
}  // anonymous namespace = internal linkage
✅ Inline functions: OK in headers
inline int clamp(int v, int lo, int hi) {
    return v < lo ? lo : v > hi ? hi : v;
}
✅ Standard pattern: declare in .h, define in .cpp
// foo.h
void publicFunction();   // declaration only

// foo.cpp
void publicFunction() { /* implementation */ }

The Rule

External linkage body in a header → MUST be inline. Otherwise: declare in header, define in .cpp.

The Key Question

"Would a client need this to USE my component?" If yes → belongs in .h. If no → belongs in .cpp.

The Compilation Tax Compounds

Every unnecessary #include in a header is a compilation tax paid by every client — including clients of clients. Moving things out of headers saves cost across the entire codebase.

Private Members Are Still In The Header

Even though clients can't access private members, they are still in the header. Changing a private type forces ALL clients to recompile.

Ch 1 — Header Discipline

Include Guards & Naming Discipline

Every header MUST have an include guard
// my_widget.h
#ifndef INCLUDED_MY_WIDGET    // INCLUDED_ prefix
#define INCLUDED_MY_WIDGET

class MyWidget { /* ... */ };

#endif  // INCLUDED_MY_WIDGET  ← comment identifies guard

Why the INCLUDED_ prefix?

Avoids collision with other macros. The closing comment is essential in long headers — you always know what that #endif closes.

External include guards in .cpp (large-scale build optimization)
// my_widget.cpp
#ifndef INCLUDED_MY_WIDGET
#include <my_widget.h>    // skip if already open
#endif

#ifndef INCLUDED_STD_STRING
#include <string>
#endif

Build-Time Payoff at Scale

With thousands of headers, avoiding even the file I/O of opening an already-seen header can measurably reduce build times. At 10,000+ headers, this matters significantly.

Ch 1 — Include Guards

The Four Component Properties

CP1: The .h is self-sufficient

The header compiles correctly on its own, without any prior #includes. It includes everything it needs.

// GOOD: my_foo.h includes what IT needs
#ifndef INCLUDED_MY_FOO
#define INCLUDED_MY_FOO
#include <my_bar.h>   // Bar used in Foo's interface
class Foo { Bar d_bar; };  // works — bar.h included above
#endif

CP2: .cpp includes its own .h FIRST

This is what tests CP1. If the header has a missing dependency, the .cpp compilation catches it immediately.

CP3: Clients never #include the .cpp

Only the .h is part of the public interface. Clients must never directly include the implementation file.

CP4: Dependencies fully determined by #include chains

There are no "hidden" dependencies. The complete dependency graph is determined entirely by #include directives. Tools can extract this automatically.

Why These Four Matter

Together, these four properties ensure every component can be compiled, tested, and analyzed independently — the foundation of scalable physical design.

Ch 1 — Component Properties

Logical Relationships & Their Physical Cost

RelationshipMeaningPhysical Effect
Is-APublic inheritanceStrong dep on base class header
Uses-In-InterfaceUsed in .h by value, ptr, refMust #include (or forward-declare)
Uses-In-ImplementationUsed only in .cpp.cpp-only include, clients unaffected

class Foo {
    Bar  d_bar;         // BY VALUE → must #include bar.h
    Baz* d_bazPtr;      // BY POINTER → forward-declare only
    void f(const Qux&); // BY REF → forward-declare only
};

Uses-In-Name-Only Is the Key

When you only store a pointer or reference, you don't need the full class definition — just a forward declaration class Bar;. This breaks compile-time coupling at no cost.

Uses-In-Implementation: Free

If a type is only used in the .cpp (not in the header), clients are completely unaffected. This is the right place for heavy dependencies.

By-Value Members Are the Most Expensive

Storing a member by value requires the full class definition in the header. Every client pays the cost of everything that member's class needs.

Change Impact Is Predictable

Change a Level 4 → nothing below Level 4 is affected.
Keep dependency cones narrow.

Ch 1 — Level Numbers

The Depends-On Relation & Implied Dependencies

What "Depends On" Means

Component Y Depends-On X if Y's .h or .cpp includes X's .h. Dependencies are transitive: if C→B and B→A, then C→A.

// Interface dependency (affects clients)
// y.h includes x.h → ALL clients of Y also pull in X

// Implementation dependency (private)
// y.cpp includes x.h → clients of Y are NOT affected

Interface vs. Implementation Dependencies

A dependency in the .h propagates to ALL clients transitively. A dependency only in the .cpp is private — clients are completely shielded from it.

Implied Dependencies — The Hidden Danger

A.h includes B.h
B.h includes C.h
→ Any file including A.h also includes C.h (transitive)

Later: B removes its #include of C.h
→ Files relying on C.h via A.h → BREAKS

The Rule

Never rely on transitive includes. Always #include what you directly use. Transitive inclusions are an accident, not a contract. They may be removed at any time.

No cycles between components, no cycles between packages, no cycles between package groups. Acyclic all the way up — without exception.

Balance Matters

The aggregation hierarchy should be roughly balanced. A package group with one giant package and many tiny ones is a design smell — it indicates poor factoring.

Why Three Levels?

Fewer than three is too coarse to manage. More than three is too complex. Three levels is enough even for the largest enterprise codebases.

Ch 2 — Packaging Hierarchy

Unit of Release (UOR): What Gets Deployed

The UOR is the Atomic Deployment Unit

A Unit of Release is the outermost physical aggregate deployed atomically. Clients either take the whole UOR — or none of it.

// Clients link against the whole UOR
libbdl.a  contains:
  bdlt_date.o  bdlt_calendar.o
  bdlb_string.o
  bdls_filesystemutil.o
  ... (all components from all packages)

Internal Structure Is Hidden

The internal package structure is an organizational and testing detail. Clients do not need to know which package a component lives in — they just link the UOR.

Three Deployment Modes

ModeFormatTrade-off
Static.aFast link, larger executables
Shared.so / .dllSmaller executables, runtime binding, ABI constraints
SourceComponent-levelMaximum portability / flexibility

Deployment Mode Is a Business Decision

Physical design must support all three modes. The choice is determined by deployment context, performance needs, and license requirements — not by the library author.

Three-Way Naming Coherence

// Three names — one consistent identity
Namespace:  bdlt
Component:  bdlt_date      // pkg + "_" + local name
Header:     bdlt_date.h

// When you see this in unfamiliar code:
bdlt::Date d;
// You IMMEDIATELY know:
→  class is  Date  in namespace  bdlt
→  declared in  bdlt_date.h
→  lives in  bdlt  package of  bdl  group
// No searching. No guessing.

The Key Rule

"The point of use should be sufficient to identify location."
— Lakos Design Rule 2.4.6

Why This Coherence Matters at Scale

In a codebase with thousands of components across dozens of packages, being able to navigate from a class name to its header to its package instantly — without a search engine — saves enormous developer time every day.

Why Opaque Prefixes Are Fine

bdlt doesn't mean anything at first glance. But after a few weeks in the codebase, it becomes immediately recognizable as the date/time package. Familiarity makes mnemonics meaningful.

Convention Is the Force Multiplier

One developer following this convention helps a little. All developers always following it means anyone can navigate any part of the codebase — even code they've never seen before.

Ch 2 — Name Cohesion

Key Naming Conventions

// Package: pkg  |  Component: pkg_date

// Header file name
pkg_date.h

// Class name — always in its package namespace
class pkg::Date { ... };

// Data member naming conventions
class Date {
    int d_year;           // d_ prefix: instance data member
    int d_month;
    int d_day;
    static int s_count;   // s_ prefix: static class member
};

Global Uniqueness Requirement

Every .h file name must be unique across the entire enterprise. This enables all headers to be installed in one flat directory. No path context needed.

Why the d_ Prefix for Members?

Immediately distinguishes member variables from local variables and parameters — critical in large methods. You always know if you're touching object state or a local.

Automated Tooling Relies on Conventions

Dependency extraction tools, package management systems, and code navigation tools all work because naming is mechanically predictable. Conventions enable tooling at scale.

The Flat Header Install

The package prefix in every filename prevents collisions when all headers are installed to one flat directory (e.g., /usr/include/). No subdirectory paths. No install-time renaming.

Ch 2 — Naming Rules

The Most Important Rule You Probably Break

❌ NEVER in header files — poisons every downstream file
using namespace std;      // VIOLATION in a .h file
using std::vector;         // also VIOLATION — leaks into every client

// A "using" in a header propagates to EVERY file that includes it,
// and every file that includes THOSE files. Transitively. Forever.
✅ OK only inside function bodies — scoped and contained
void foo() {
    using std::cout;    // scoped — doesn't affect anything outside
    cout << "hello\n";
}

The Scale Problem

In a large codebase, using namespace X in a header means:
• Name collisions become unpredictable
• You can't identify which namespace a symbol came from
• Removing the directive later breaks all downstream code

Prefer <iosfwd> over <iostream> in Headers

// ✅ Just a forward declaration — minimal cost
#include <iosfwd>
std::ostream& operator<<(std::ostream& os, const Date& d);
// ❌ Don't pull in full <iostream> just for a declaration

The Payoff

Clean header files = fast builds + no mystery name collisions + headers that remain composable.

Standard Component Source Layout

pkg_mytype.h — the header (contract)
// pkg_mytype.h                        -*-C++-*-
// CLASSES:
//   pkg::MyType - a brief description
#ifndef INCLUDED_PKG_MYTYPE
#define INCLUDED_PKG_MYTYPE

#ifndef INCLUDED_STD_STRING

#endif

namespace pkg {

    std::string d_name;
  public:
    explicit MyType(const std::string& name);
    const std::string& name() const;

}  // close namespace pkg
#endif  // INCLUDED_PKG_MYTYPE
pkg_mytype.cpp — the implementation
#ifndef INCLUDED_PKG_MYTYPE  // external guard
#include <pkg_mytype.h>      // ← ALWAYS first
#endif

namespace pkg {
MyType::MyType(const std::string& name)
: d_name(name) {}

const std::string& MyType::name() const {
    return d_name;
}
}  // close namespace pkg

Every Detail Is Intentional

File-mode comment, class list comment, external guard on every include, namespace close comment, include guard comment — each element exists to help humans and tools navigate at scale.

This Layout Is Machine-Parseable

Automated tools can extract class names, dependencies, and ownership from file headers. Consistency is what enables tooling to work reliably across millions of lines of code.

Ch 2 — Package Groups

The Most Important Consequence of Acyclic Design

The Requirement

Every component must be independently unit-testable using only components at strictly lower levels.

Test math_util  (L1): links nothing internal
Test date       (L2): links math_util only
Test calendar   (L3): links math_util + date
Test cal_cache  (L4): links math_util + date + calendar

// Test at each level proves that level in ISOLATION.
// A failure at L2 cannot be caused by an L3 bug.

The Pyramid of Confidence

Each test only requires confidence in components already proven by lower-level tests. Bug isolation becomes trivial: if L1 passes and L2 fails, the bug is in L2.

What Cycles Destroy

If A depends on B AND B depends on A:
→  Cannot test A without linking B
→  Cannot test B without linking A
→  "Unit tests" become integration tests
→  Tests are slow, flaky, hard to diagnose
→  Developers stop writing them

This Is Not a Preference

Acyclic dependencies are not a code style preference. They are a hard prerequisite for fine-grained unit testing. You cannot have one without the other.

Ch 2 — Hierarchical Testability

Asking the Right Question When Adding Code

Not Just: "What Class Should I Create?"

Physical thinking asks: "Where should this code physically live?" — which level, which package, which component.

// When you write this:
#include <calendar.h>

// You also pull in:
→  date.h
→    math_util.h
→      <cstdint>

// You are now bound to ALL of these at compile time.
// Your recompile set just grew by every client of these headers.

The Dependency Cone

Every header you include pulls in a cone of transitive dependencies. Keeping that cone narrow is the core job of physical design.

Good Physical Citizenship

  • Find what already exists before creating something new
  • Place new code at the lowest level that satisfies requirements
  • Never reach upward — lower levels must not depend on higher levels
  • Forward-declare instead of including where possible
  • Keep headers minimal — only what the interface requires

The Question to Ask Every #include

"Do I really need this here — in the header? Or can this move to the .cpp?"

Physical Thinking Is a Habit

The best physical designs come from developers who internalize these questions and ask them automatically with every new piece of code.

Ch 2 — Thinking Physically

A Critical Design Decision: Primitive vs. Non-Primitive

Primitive Operation

NEEDS access to private state to be implemented efficiently, OR is logically central to what the abstraction represents.

Non-Primitive Operation

COULD be implemented using only the public (primitive) interface. Does not need private access.

class Polygon {
  public:
    void addVertex(double x, double y); // PRIMITIVE
    double area() const;               // PRIMITIVE
    int vertexCount() const;           // PRIMITIVE
    // ❌ Non-primitives — should NOT be here:
    bool isConvex() const;
    void rotate(double angle);
};

Non-Primitives Belong in a Utility

// polygon_util.h — separate component
struct PolygonUtil {
    static bool  isConvex(const Polygon& p);
    static void  rotate(Polygon* p, double angle);
    static bool  isRegular(const Polygon& p);
};
// Users who don't need these → don't link this .o

Why the Separation Matters

  • Smaller, more stable core component
  • Clients who only need area() don't pull in geometry utilities
  • Add utilities without touching Polygon
  • Test core and utilities independently
Ch 3 — Primitive vs Non-Primitive

The Benefits of Separating Non-Primitive Functionality

1. Smaller Compile-Time Footprint

Clients who only need area() don't compile isConvex()'s dependencies — potentially heavy geometry math libraries.

2. Simpler, More Stable Core

Fewer reasons for Polygon to change → fewer rebuilds for its clients. A stable primitive interface is worth its weight in gold.

3. Independent Extensibility

Add new utilities without touching the core component at all. This is the Open/Closed Principle applied physically.

4. Easier, Independent Unit Testing

Test Polygon with 20 test cases. Test PolygonUtil separately with its own test cases. No entanglement. Test failures localize immediately.

5. Explicit Link-Time Footprint

Users who don't need PolygonUtil don't link it. The linker only pulls in .o files you reference. Binary size and link time stay proportional to actual use.

The Scale Effect

In a small codebase, stuffing everything in one class is fine. At scale, every unnecessary dependency multiplies. Separation pays dividends that grow super-linearly with codebase size.

Keep classes in separate components. Colocation requires explicit justification.

Ch 3 — Colocation Rules

Cyclic Dependencies: Why They Are Fatal

// The cycle:
Component A #includes Component B
Component B #includes Component A

Immediate Consequences

  • Cannot compile A without B; cannot compile B without A
  • Cannot test A without linking B; cannot test B without A
  • Level numbers undefined for both — graph is not a DAG
  • Any change to A's header triggers B's recompile and vice versa

Long-Term Consequences at Scale

  • One cycle always becomes many — developers copy the pattern
  • Integration testing replaces unit testing
  • Rebuild times grow to minutes for any change
  • Code cannot be extracted and reused elsewhere
  • Refactoring becomes impossibly risky

Why Cycles Spread

Once a cycle exists in a codebase, it normalizes the pattern. Developers see it and add more. Each new cycle makes the system harder to test, understand, and change. They compound.

The Cancer Analogy

Cyclic dependencies are like cancer in a codebase. They start small, grow silently, and eventually everything is connected to everything. Treatment becomes harder the longer you wait.

The Rule Is Absolute

There is no acceptable cyclic dependency at any level in the physical hierarchy. Not between components. Not between packages. Not between package groups.

Ch 3 — Levelization Techniques

Technique 1: Classic Levelization — Cut the Reverse Dependency

The Scenario

Manager and Employee seem to have a bidirectional relationship: Manager holds Employees, Employee knows its Manager.

// Before (cycle):
Manager ←→ Employee
// Manager holds Employees; Employee knows Manager
// Neither can be compiled or tested without the other

// After (acyclic):
Manager → Employee
// Manager holds Employees
// Employee knows NOTHING about Manager

The Key Insight

Many "bidirectional" logical relationships can be implemented with a unidirectional physical dependency. One direction is often merely convenient, not required.

The Question to Ask

"Does Employee really need to know its Manager at this level?"

  • For HR operations (probably) — put that at a higher level
  • For the core Employee data model — almost certainly not

Ask: "What Does Each Side REALLY Need?"

Rigorously examine what each side truly requires from the other at this level of abstraction. Often the reverse direction disappears under scrutiny.

The Most Underused Technique

Many developers add the reverse dependency out of habit. Classic levelization is free — it just requires questioning the assumption that both directions are needed.

Technique 2: Escalation — Add a Third Component Above Both

The Scenario

EventQueue processes Event objects, but Event needs to be put into queues. Making them mutually dependent creates a cycle.

// Before: attempt at mutual dependency
EventQueue ←→ Event   // cycle — neither tests alone

// After: escalated to a third component
EventManager (L3) → EventQueue (L2)
EventManager (L3) → Event      (L1)
EventQueue   (L2) → Event      (L1)
// Event knows NOTHING about queues or managers

What EventManager Does

  • Creates Events
  • Puts them in EventQueues
  • Processes them
  • Both Event and EventQueue stay clean and focused at lower levels

The Principle

If two things need to talk, don't force them to talk directly. Build a room above them that handles the conversation. The interaction belongs at a higher level.

When to Use Escalation

When two otherwise-clean components need to cooperate and neither should know about the other. The "knowledge" of their cooperation lives at the level above.

Technique 3: Demotion — Move Low-Level Code Down

The Scenario: The "Big Ball of Mud" Utility Module

A utility module has grown to contain capabilities that belong at completely different levels:

// all_utils (Level 5) — contains everything
→  string hashing    (should be Level 1)
→  date parsing      (should be Level 3)
→  report generation (should be Level 5)

// Level 1 code that needs string hashing
// must link against Level 5!
// After demotion — each capability at its natural level
str_hash  (L1)  ← just the hash function
date_util (L3)  ← date parsing
report    (L5)  ← report generation

// Level 1 components link ONLY str_hash (Level 1)
// No more dragging in Level 5 to get Level 1 work done

How to Identify Demotion Candidates

Scan each component for functionality that could be self-contained at a lower level. If a function only uses stdlib and no higher-level types, it belongs lower.

The Blob Refactoring Strategy

Start by identifying the lowest-level capability in the blob and demote it first. Repeat until each component contains only code that naturally belongs at that level.

Ch 3 — Demotion

Technique 4: Opaque Pointers (PIMPL)

❌ Before — strong compile-time dependency
// device.h
#include <heavy_driver.h>  // every Device client pays this!
class Device {
    HeavyDriver d_driver;   // full type required in header
};
✅ After — opaque pointer breaks the coupling
// device.h — only needs <memory>
class DeviceImpl;          // forward declaration — no include
class Device {
    std::unique_ptr<DeviceImpl> d_impl_p;
  public:
    Device();
    ~Device();     // must be DEFINED in .cpp
};
device.cpp — ONLY this file pays the cost
#include <device.h>
#include <heavy_driver.h>   // cost isolated here
class DeviceImpl { HeavyDriver d_driver; };
Device::Device() : d_impl_p(std::make_unique<DeviceImpl>()) {}
Device::~Device() = default; // DeviceImpl complete here ✓

The Critical PIMPL Rule

The destructor (and any copy/move operations) MUST be declared in the header and defined in the .cpp — where DeviceImpl is complete. Defaulting in the header = incomplete type error.

BenefitCost
Clients don't recompile when impl changesOne heap allocation per object
ABI stabilityIndirect dispatch through pointer
Smaller header, fewer cascading includesCan't inline through impl

When to Use PIMPL

Long-lived objects, shared library APIs, implementations that will change. Not for short-lived value types in hot paths.

Ch 3 — PIMPL

PIMPL: Gotchas and Trade-offs in Detail

❌ Wrong — defaulting destructor in header = incomplete type error
class Device {
    std::unique_ptr<DeviceImpl> d_impl_p;
  public:
    ~Device() = default;  // DeviceImpl INCOMPLETE → error!
};
✅ Correct — declare in header, define in .cpp
// device.h
class Device {
  public:
    ~Device();      // declared only — body in .cpp
    Device(Device&&); // same rule for move
};
// device.cpp
Device::~Device() = default;   // DeviceImpl COMPLETE here ✓
Device::Device(Device&&) = default; // same

The Same Rule Applies To:

  • Copy constructor (if needed)
  • Copy assignment operator
  • Move constructor
  • Move assignment operator
  • Any function that touches the impl type

Trade-off Summary

Use PIMPL when: Impl may change, ABI stability matters, it's a long-lived object, or it's a shared library API.

Don't use PIMPL when: Value type used in tight loops, performance-critical, or the type is rarely recompiled anyway.

ABI Stability Is the Killer Feature

A shared library using PIMPL can ship a new implementation without any client needing to recompile. The binary interface is defined entirely by the opaque pointer — nothing about the impl is exposed.

Technique 5: Dumb Data — Structure Without Semantics

The Scenario

A Node stores a type tag. Graph needs to interpret it. If Node imports graph-level semantics, it creates an upward dependency — a cycle.

// ✅ DUMB DATA — Node stores structure, no logic
struct Node {
    int d_type;   // just a tag — no interpretation here
    int d_value;
};
// Graph at a higher level provides interpretation
class Graph {
    void processNode(const Node& node) {
        switch (node.d_type) {
            case NODE_TYPE_SOURCE: /* ... */ break;
            case NODE_TYPE_SINK:   /* ... */ break;
        }
    }
};

The Key Separation

The low-level type stores structure (raw data, tags, identifiers). The higher-level component provides semantics (interpretation, behavior, policy).

Where This Pattern Appears

  • AST nodes without knowing the interpreter
  • Event structs without knowing the handler
  • Message headers without knowing the protocol logic
  • Configuration records without knowing how they're applied

Why It Works

The low-level type becomes completely reusable — it carries no knowledge of the domain it will be used in. Multiple higher-level components can assign completely different semantics to the same data.

Ch 3 — Dumb Data

Technique 6: Callbacks — Let the Caller Register Behavior

The Problem Without Callbacks

A lower-level component needs to "report" or "notify" something upward. Without callbacks, it must #include a higher-level type — creating an upward dependency.

// 1. Function pointer + void* userdata — simplest
using DataCb = void(*)(const char* msg, void* userData);

// 2. Plain function pointer — no state
using Handler = void(*)(int eventType);

// 3. std::function — stateful, captures lambdas
std::function<void(int)> onEvent;

// 4. Protocol (abstract interface) — most powerful
class EventHandler {
  public:
    virtual ~EventHandler() = default;
    virtual void onEvent(int type) = 0;
};

The Direction-Reversal Principle

Instead of the low-level component reaching up to call a specific type, the high-level component registers its behavior with the low-level component at setup time.

Choosing the Right Flavor

  • Function pointer + void* — C-compatible, zero overhead, manual state
  • std::function — capture lambdas, more overhead, modern C++
  • Protocol class — most powerful, virtual dispatch, enables full DI

Choose the simplest one that satisfies the requirement.

The Result

The low-level component knows only about the interface (callback signature or protocol class). It has zero knowledge of who will call it or what the caller does with the notification.

Ch 3 — Callbacks

Callbacks: The Protocol Form — The Most Powerful

Level 1 — Pure Abstract Interface (the protocol)

class Serializer {
  public:
    virtual ~Serializer() = default;
    virtual void write(int value) = 0;
    virtual void write(const std::string& s) = 0;
};

Level 2 — Date Uses Serializer

No knowledge of any concrete implementation.

class Date {
    int d_year, d_month, d_day;
  public:
    void serialize(Serializer& s) const {
        s.write(d_year); s.write(d_month); s.write(d_day);
    }
};
// Level 3+: Concrete serializers at higher levels
class JsonSerializer   : public Serializer { /* ... */ };
class BinarySerializer : public Serializer { /* ... */ };

// Application wires it together at the top level
JsonSerializer js;
Date d(2024, 1, 15);
d.serialize(js);  // Date never knew about JSON

The Dependency Arrow Points Downward

Date depends on Serializer (abstract, at Level 1).
Date does NOT depend on JsonSerializer (concrete, Level 3+).
The application wires the concrete to the abstract at the top.

The Result

Add a new XmlSerializer at Level 3+ without touching Date or Serializer. Zero recompilation of lower-level components. Infinite extensibility.

Ch 3 — Protocol Callbacks

Technique 7: Protocol Class — Dependency Inversion, Lakos Style

The Problem

DataStore must persist data. Hardcoding SqliteDB makes DataStore useless without SQLite — and untestable without a database.

// ✅ Protocol class — pure abstract, no implementation
class PersistenceAdapter {
  public:
    virtual ~PersistenceAdapter() = default;
    virtual void save(const std::string& key, int val) = 0;
    virtual int  load(const std::string& key) = 0;
};

// Level 2: DataStore knows ONLY the abstract interface
class DataStore {
    PersistenceAdapter& d_adapter;
  public:
    explicit DataStore(PersistenceAdapter& a) : d_adapter(a) {}
    void store(const std::string& k, int v) { d_adapter.save(k, v); }
};
// Level 3+: Each concrete impl in its own component
class MemoryPersistence : public PersistenceAdapter { /* ... */ };
class FilePersistence   : public PersistenceAdapter { /* ... */ };
class SqlitePersistence : public PersistenceAdapter { /* ... */ };

Physical Effect

DataStore's header includes only PersistenceAdapter's header. Clients of DataStore have zero knowledge of SQLite, files, or memory backing — unless they use those components directly.

What You Can Do

  • Swap persistence backend without touching DataStore
  • Test with MemoryPersistence in microseconds
  • Write DataStore before the backend exists
  • Multiple backends simultaneously
Ch 3 — Protocol Class

Protocol Class: Swap Without Recompiling

// Tests: fast, zero I/O
MemoryPersistence memStore;
DataStore testStore(memStore);   // microseconds

// Production: just swap the adapter
FilePersistence fileStore("/var/data/prod.dat");
DataStore prodStore(fileStore);  // same DataStore!

// Upgrade to SQLite later — DataStore never changes!
SqlitePersistence dbStore("prod.db");
DataStore newStore(dbStore);     // zero code change

No Recompilation of DataStore

Adding SqlitePersistence does not recompile DataStore, PersistenceAdapter, or any of their clients. Only the new concrete component is compiled.

This Is Dependency Inversion (DIP)

High-level modules should not depend on low-level modules. Both should depend on abstractions. The protocol class IS the abstraction. Physical design makes the DIP possible at scale.

Design vs. Patterns

In OOP terms, this is the Strategy Pattern. In physical design terms, it's just a protocol class at Level 1 with concrete implementations at higher levels. The pattern is the same — physical design makes it work at enterprise scale.

The Test Value Is Enormous

A suite of 10,000 tests that each create a DataStore with MemoryPersistence runs in seconds. With FilePersistence, it might take hours. Protocol classes are what make fast test suites possible.

Avoiding Excessive Link-Time Dependencies

The Date Class That Got Too Fat

A Date class should provide primitive date operations. But "business day" requires calendars, locale data, and market-specific rules — none of which belong in Date.

// ❌ BAD: business day logic IN Date
class Date {
  public:
    bool isBusinessDay(const Calendar& cal) const;
    // requires Calendar.h — every Date user links Calendar!
};

// ✅ GOOD: demoted to utility
// bdlt_date.h
class Date { /* pure date arithmetic only */ };

// bdlt_dateutil.h (separate component)
struct DateUtil {
    static bool isBusinessDay(const Date& d, const Calendar& c);
};

The Rule

A component's link-time footprint should match its conceptual footprint. If a client only needs Date, they should not be forced to link Calendar.

The Gradual Degradation Pattern

A clean Date at Level 2 slowly has business logic added. One year later, it's at Level 7. Nobody can use it without the entire finance stack linked in. Every new feature made it worse — gradually, then all at once.

Link-Time Footprint Is a Design Constraint

Ask of every method: "Should users of this component be required to link everything this method requires?" If no, it belongs in a utility component.

Lateral vs. Layered Architectures

✅ Layered (vertical) — preferred
Level 5:  ├─ A
Level 4:  ├─ B
Level 3:  ├─ C
Level 2:  ├─ D  E
Level 1:  └─ F  G  H  I
→ Each level only depends on lower levels
→ CCD is low → fast incremental builds
→ Change F → only L2+ affected

The Tower Analogy

A layered architecture is like building a tower. Each floor rests on those below it. You can inspect, repair, or replace a floor independently without touching the others.

❌ Lateral (horizontal) — antipattern
Level ?:  A ←→ B ←→ C ←→ D ←→ E
              ↑↓  ↑↓  ↑↓
→ Everyone knows everyone
→ CCD is high → slow builds
→ Change anything → everything rebuilds

The Web Analogy

A lateral architecture is like a spider web. Cut one strand and everything else shifts. Every node is load-bearing for the whole system. Nothing can be safely isolated.

Why Lateral Architectures Form

Developers add convenience dependencies — "I just need one function from over there." Over time, every component has a few references to every other. Level numbers become impossible to assign.

Ch 3 — Architecture Patterns

Cumulative Component Dependency (CCD): Measuring Architectural Health

Definition

For each component, count how many components it transitively depends on (including itself). Sum all counts. That sum is the CCD.

// Layered (5 components: A→B→C→D→E)
A: 1   (itself only)
B: 2   (B + A)
C: 3   (C + B + A)
D: 4   (D + C + B + A)
E: 5   (E + D + C + B + A)
CCD = 15

// Lateral (all 5 know all 5)
Each = 5, total × 5 components
CCD = 25  ← 67% worse, same 5 components

// At scale — difference is QUADRATIC
100 layered:  CCD ≈ 5 050
100 lateral:  CCD = 10 000

What CCD Measures

CCD is the total "recompile cost" across the whole system if one component at each level changes. Low CCD = small recompile sets = fast incremental builds.

CCD as a Health Metric

Track CCD over time. If it grows faster than line count, you are accumulating physical design debt. The architecture is becoming more lateral — more interconnected — even if no single change looks significant.

The Practical Use

Compute CCD before and after a refactoring. If it goes down, you improved the architecture. If it goes up, you made it more coupled. CCD is a quantitative signal, not just a feeling.

Avoiding Inappropriate Physical Dependencies

Don't Bet on a Single Technology

A library component that hardcodes a dependency on MySQL forces ALL users of that library to install MySQL — even if they never touch the database code.

// ❌ BAD: library depends on specific database
class UserRepository {
    MySqlConnection d_conn;  // requires mysql/mysql.h
  public:
    User findById(int id);
};

// ✅ GOOD: library depends on abstract interface
class UserRepository {
    UserPersistence& d_store;   // abstract
  public:
    explicit UserRepository(UserPersistence& store);
    User findById(int id);
};
// MySqlUserPersistence lives in a separate, higher-level component

The Antipattern Catalog

  • #ifdef BUYSIDE_ONLY — domain logic in library code
  • Application-specific logic embedded in reusable components
  • Accessing global/static state from library code
  • Hiding potentially reusable code inside application components
  • Including heavyweight platform headers in public interfaces

The General Rule

Library components must be policy-free. They should provide mechanism, not policy. Policy (which database, which format, which locale) belongs in application-level components or in wiring at the top of the hierarchy.

The Portability Test

Ask: "Could I use this component in a completely different application, for a completely different customer, on a completely different platform?" If not, it has inappropriate dependencies.

Ch 3 — Inappropriate Dependencies

Unnecessary Compile-Time Dependencies

The Surprising Fact

Private member variables are fully visible in the header — even though clients can't access them. Changing a private member forces ALL clients to recompile.

// ❌ HEAVY header — private impl exposed
class Widget {
  private:
    LargeDatabase d_db;     // clients see this
    ComplexCache  d_cache;  // clients must include
    NetworkConn   d_conn;   // large_database.h etc.
  public:
    void doThing();
};
// ✅ INSULATED header — impl hidden
class Widget {
    struct Impl;
    std::unique_ptr<Impl> d_impl_p; // only <memory> needed
  public:
    void doThing();
};
// Impl's details live entirely in widget.cpp

Compile-Time vs. Link-Time Coupling

Coupling TypeControlled ByTool
Compile-timeWhat appears in your .h (even private!)PIMPL, forward declarations
Link-timeWhat you call at runtimeProtocol class, factoring

Encapsulation ≠ Physical Hiding

OOP encapsulation hides data from language-level access. Physical insulation hides data from the compiler. They are complementary but completely different mechanisms — both matter.

The Impact at Scale

In a 10,000-file codebase, adding one heavy private member to a widely-used class can trigger a 20-minute rebuild. The PIMPL idiom converts that to a 30-second incremental build.

Ch 3 — Compile-Time Coupling

Three Approaches to Total Insulation

1. Pure Abstract Interface (Protocol Class)

Best for runtime polymorphism & multiple implementations. Trade-off: virtual dispatch overhead.

class Renderer {
  public:
    virtual void draw(Shape*) = 0;   // abstract
    virtual ~Renderer() = default;
};
// OpenGLRenderer lives elsewhere — fully hidden from clients

2. Fully Insulating Concrete Wrapper (PIMPL)

Best for wrapping a component you can't change. Trade-off: forwarding all API calls.

class Widget {               // public API — minimal header
    std::unique_ptr<WidgetImpl> d_impl_p;
  public:
    void doThing();  // forwards to d_impl_p->doThing()
};

3. Procedural Interface (C-compatible)

Best for shared libraries, cross-language ABI, maximum stability. Trade-off: verbose, manual resource management.

// C-compatible procedural interface
extern "C" {
    void* widget_create();
    void  widget_doThing(void* widget);
    void  widget_destroy(void* widget);
}

All Three Serve the Same Goal

Hide the implementation details from the compiler — not just from the language. The choice depends on polymorphism needs, performance constraints, and ABI stability requirements.

Composable

These techniques can be combined. A protocol class can use PIMPL internally. A C interface can delegate to a protocol class. Use what the situation requires.

When to Insulate and When Not To

SituationInsulate?Reason
Short-lived value types (Point, Rect)❌ NoHeap allocation overhead not worth it
Long-lived objects, complex implementations✅ YesImpl can change without client recompile

Rule of Thumb

If your header is heavy AND the implementation is likely to change → insulate.
If your type is simple and stable → don't pay the overhead.

The ABI Special Case

For shared library public APIs, insulation is not optional — it's mandatory. Without it, any change to a private member requires recompiling all consumers, defeating the entire purpose of a shared library.

Think About Who Your Consumers Are

Internal library (same team, same repo) — insulation optional. Public API (external teams, separate release) — insulate everything that might change.

The Component as the Unit of Design

Traditional OOP Mindset

"I need a Calendar class with methods for business day calculations, formatting, and parsing."

Component-Based Physical Design Mindset

"I need:

  • A bdlt_calendarutil component (algorithms)

Not "how many classes?" but "how many components, at what levels?" A design with 5 classes in one component is often worse than one with 10 classes across 5 focused components.

A Clean Example

// 50 classes, 1 cycle → cannot test, cannot reuse

vs.

// → maintenance is proportional to the part changed

The Component Is the Atom

Just as chemistry is built from atoms that cannot be divided without losing their properties, software at scale is built from components that cannot be split without losing testability and reusability.

The Unit of Design, Testing, and Deployment

The component is the unit at all three levels. Design one component at a time. Test one component at a time. Deploy a collection of components as a UOR.

Advanced: Subpackages and Subordinate Components

Subpackages

Large packages can be organized into subpackages to manage internal complexity. These are an organizational tool — clients still see the package-level interface. The package prefix still governs.

Subordinate Components

A component sometimes needs a helper that is NOT part of the public API. A subordinate component handles implementation details and is not installed as part of the public interface.

bdlt_date.h          ← public component
bdlt_date.cpp
bdlt_date_impl.h     ← subordinate: internal, NOT installed
bdlt_date_impl.cpp

Component-Private Classes

Small helpers that are entirely private to one component can be forward-declared in the header (so the compiler knows the name) and fully defined in the .cpp:

// date.h
class Date_Impl;        // declared — clients see the name
class Date {
    Date_Impl* d_impl_p; // cannot be used by clients
};

// date.cpp — full definition here only
class Date_Impl {
    /* all the messy internals */
};

The Rule Still Holds

Even with subpackages and subordinate components, the package prefix convention, acyclic dependency rule, and testability requirement all apply unchanged. The patterns scale.

Ch 3 — Advanced Organization

Integrating Code That Doesn't Follow the Methodology

The Challenge

Legacy, third-party, or open-source code won't follow your naming conventions, include discipline, or cyclic dependency rules. The answer: wrap it.

The Wrapper Pattern

Create a conforming component that presents a clean interface, hiding the third-party header from all your clients:

// thirdparty_json_wrapper.h — YOUR clean interface
class JsonDocument {
  public:
    bool parse(const std::string& text);
    std::string getString(const std::string& key) const;
    // NO nlohmann::json visible in this header!
};

// thirdparty_json_wrapper.cpp
#include <thirdparty_json_wrapper.h>
#include <nlohmann/json.hpp>  // hidden — clients NEVER see this

Why One Wrapper Component?

If you need to swap nlohmann::json for RapidJSON, you change exactly ONE component. All your clients remain untouched. The isolation makes the swap possible.

Legacy Code Strategy

  • Wrap legacy modules — don't touch them
  • Introduce naming conventions at the boundary
  • Gradually extract reusable parts into conforming components
  • Use the wrapper as your strangler-fig boundary

The Strangler Fig Pattern

Name comes from the tree that grows around an existing tree. Wrap the legacy component, slowly replace its functionality with proper components, then remove the original.

Ch 3 — External Code Integration

From Development to Deployment: The Value of Unique Names

Global Uniqueness Enables Everything

Every .h and .o file has a globally unique name across the enterprise. This single rule unlocks:

  • Flat deployment: all headers in /usr/include, no collisions
  • Automated tooling: find any header without knowing paths
  • Enterprise grep: rename a component — exactly one match
  • Flexible builds: source and install directories can differ

Three Deployment Modes

ModeFormatTrade-off
Static.aFaster link, larger executables
Shared.so/.dllSmaller binary, runtime binding, ABI constraints
SourceComponent-levelMax portability, recompiled per consumer

Deployment Mode Is a Business Decision

The choice between static, shared, and source distribution is determined by deployment context, performance needs, and license requirements. The physical design must support ALL three modes — not just the one you happen to need today.

Why This Matters to the Developer

If your design only works as a static library (e.g., because it relies on internal linkage tricks), you've limited your future options. Good physical design keeps all doors open.

Testability = Deployability

A codebase where every component is independently testable can also be independently deployed. The two properties are two sides of the same coin: acyclic, hierarchical design.

Without Metadata

Dependencies exist only implicitly in #include chains. You can inspect them, but you cannot enforce them automatically. Rules are suggestions, not constraints.

With Metadata

The build system can verify that actual dependencies respect declared constraints. If someone adds an unauthorized #include, the build fails. Rules become enforceable policy.

Metadata Is Policy as Code

The .dep file is an executable specification of your architecture. Every build validates the architecture. Violations are caught automatically, immediately, by tooling — not discovered months later in a code review.

Ch 3 — Metadata

How to Diagnose Physical Design Problems

Ask in This Order

1. "Is there a cycle here?" → Assign level numbers. Can't assign? You found a cycle. 2. "Can I test this in isolation?" → If not, there's a hidden dependency. Find it. 3. "Why is the build so slow?" → Check what's in the headers. CCD gives the number. 4. "Why can't I reuse this?" → Non-primitive functionality in core component, or → inappropriate deps at the wrong level. 5. "Why did changing X break Y?" → Transitive include chain. Implied dependency. → Should have used forward decl or PIMPL.

Tooling That Helps

# Show ALL transitive includes of a file
cpp -M file.cpp

# Show all symbols defined in an object file
nm -C binary.o

# Bloomberg's static analysis tool
bde_verify component.cpp

# Scan for include-what-you-use violations
iwyu component.cpp

The Most Valuable Diagnostic

Run cpp -M on a component you think is simple. Count the total number of headers transitively included. If it's 50+ for a 100-line file, you have a physical design problem — not a code quality problem.

CCD Is Your Ongoing Dashboard

Compute CCD weekly. Track it in version control as a team metric. Rising CCD = declining architecture quality. Falling CCD = genuine improvement. Numbers don't lie.

Appendix — Diagnostics

The Three Golden Rules

Rule 1: Acyclic Dependencies

No cycles in the physical dependency graph — between components, packages, or package groups. If you cannot assign a level number to a component, you have a violation.

Rule 2: Hierarchical Testability

Each component can be tested independently using only lower-level components. If your test driver requires components at the same or higher level, your design is wrong.

Rule 3: Minimal Coupling

Expose only what clients need. Hide everything else in .cpp files. Track CCD. Apply PIMPL or protocol classes when headers become heavy.

Everything Reduces to These Three

Cycle? → Use one of the 7 levelization techniques Can't test? → Find and cut the hidden dependency Build too slow? → Reduce header weight, reduce CCD Can't reuse? → Separate non-primitive functionality ABI breaks? → Apply insulation (PIMPL or protocol class)

These Rules Are Not Style — They Are Structural

You cannot partially follow them. A codebase with "mostly" acyclic dependencies accumulates cycles over time, because one cycle normalizes the pattern. All three rules must be applied consistently and enforced by tooling.

Appendix — Golden Rules

Quick Reference: Design Rules Checklist

Component Properties (CP)

  • ☐ CP1: Header is self-sufficient — compiles standalone
  • ☐ CP2: .cpp includes its own header FIRST
  • ☐ CP3: No non-header content exported from .cpp
  • ☐ CP4: Dependencies fully determined by #include chains

Naming & Design Rules (DR)

  • ☐ DR-1: No cyclic physical dependencies
  • ☐ DR-2: Filename = package prefix + _ + local name
  • ☐ DR-3: Header names unique enterprise-wide
  • ☐ DR-4: Members prefixed d_, statics prefixed s_
  • ☐ DR-5: No using namespace in headers
  • ☐ DR-6: No using declarations in headers (outside function scope)
  • ☐ DR-7: Free operators declared in same namespace as the type
  • ☐ DR-8: Use <iosfwd> not <iostream> in headers

Architecture Rules

  • ☐ Every component has an assigned level number
  • ☐ Test drivers link only lower-level components
  • ☐ Non-primitive functionality lives outside core component
  • ☐ No using namespace directives in any header
  • ☐ CCD tracked and not growing faster than LOC

Code Review Questions

  • Does this new #include belong in the .h or the .cpp?
  • Is this operation primitive or non-primitive?
  • Does this create a cycle? What is the level number?
  • Can I write a test for this component with nothing above it linked?

Print This and Use It

The checklist is most valuable when used consistently in code review. A team that asks these questions on every PR builds a codebase that stays healthy over years.
